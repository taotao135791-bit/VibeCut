规划重构工具调用和系统prompt



问题：工具调用中，假设使用edit_clips，模型对于选择何种操作（add,update,move）表现的很差，模型在选择工具有思考，但是选择何种操作没有思考

解法：拆分工具调用，禁止出现一个大而全的工具调用
edit_clips-》 拆为 add_clips、update_clips、delete_clips、move_clips，每个仍支持 batch（接收数组），但不再混合不同操作类型 



问题：系统prompt 应当指导模型如何思考，而不是告诉它思考什么。

差的做法（当前做法）：指导模型在某类问题下先调a,后调b,规定xxx工具是干什么的

好的做法：仅供参考，禁止按照参考实现
```
# 工作原则 (Principles)
1. **谋定而后动 (Plan Upfront)**：在调用任何工具之前，你必须先在 <thinking> 中分析用户的真实意图，，并权衡哪一个最适合当前上下文。
2. **渐进式验证 (Verification Loop)**：调用工具获得结果后，先反思结果是否满足预期，如果不满足，请分析原因并尝试其他操作类型。

# 思考范式限制
每次回复时，严格按照以下结构：
<thinking>
1. 用户的核心目标是什么？
2. 当前可用的操作类型有哪些？各自的优缺点是什么？
3. 我决定选择操作类型 X，原因是...
</thinking>[执行工具调用]
```
